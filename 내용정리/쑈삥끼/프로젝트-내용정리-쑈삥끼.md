- **프로젝트 주제** — Pinky Pro 로봇 2대가 미니어처 마트에서 고객을 따라다니는 스마트 쇼핑 카트. Open-RMF 없이 자체 Fleet Router로 다중 로봇을 조정.
- **기획 배경 / 문제** — 장바구니/카트를 직접 끌어야 하는 쇼핑의 불편, 무거운 짐. "따라오는 카트 + 상품 안내 + 자동 결제"로 해결.
- **프로젝트 목표** — ① 주인 인형만 골라 추종, ② 자연어로 상품 위치 안내, ③ QR 장바구니·결제, ④ 자동 충전소 복귀까지 end-to-end 데모.
- **핵심 기능 요약** — 로그인 → 추종 → 상품 가이드(LLM) → 장바구니 → 대기 → 결제 → 복귀 (7단계).
- **전체 동작 흐름** —
  `LCD QR 로그인 → 인형 등록 → IDLE→TRACKING(추종) → 상품검색(LLM+Nav2 안내) → QR 장바구니 담기 → WAITING(자유 이동) → 결제구역 진입 시 결제 → RETURNING(빈 충전슬롯 자동 탐색) → 도크 진입 → 세션 종료`

## 2-2. 내가 구현한 기능 (git 근거)

**기능 1 — 대기(WAITING) 상태 흐름 & 카운트다운**
- 무엇: TRACKING→WAITING 전환 후 사용자 자유 이동, 타임아웃 시 자동 복귀. 프론트에 카운트다운 표시.
- 구현/기술: `bt_runner.py` WAITING 처리, `cmd_handler.py`, `config.py` `WAITING_TIMEOUT` 중앙화, `socket.js` 카운트다운·복귀 fallback.
- 선택 이유: 타임아웃 값이 여러 컴포넌트에 흩어져 불일치 → **단일 소스로 중앙화**해 동작 일관성 확보.
- 관련 파일: `shoppinkki_core/bt_runner.py`, `.../config.py`, `customer_web/static/js/socket.js`
- 수치: 대기 타임아웃 = **300초**(config), 대기 루프 10Hz(문서). 관련 커밋: "centralize WAITING_TIMEOUT", "reset WAITING timeout reliably on re-entry".

**기능 2 — 미결제 잠금(LOCKED) → 복귀(RETURNING) 상태 전환 규칙**
- 무엇: 미결제 카트는 LOCKED로 잠근 뒤 복귀시키고, 스태프 해제 전엔 세션 종료 안 됨.
- 구현/기술: `state_machine.py` 전환 테이블 — **LOCKED 진입은 WAITING에서만 허용**, RETURNING은 GUIDING/SEARCHING에서도 relay.
- 선택 이유: 임의 상태에서 LOCKED 진입 허용 시 상태머신 무결성 붕괴 → 진입 소스를 제한해 안전성 확보.
- 관련 파일: `shoppinkki_core/state_machine.py`, `control_service/robot_manager.py`
- 수치: SM 10개 상태. 커밋: "restrict LOCKED transitions to WAITING state only", "refine state machine transitions for LOCKED and RETURNING".

**기능 3 — control_service 세션 생명주기 관리**
- 무엇: REST 세션 생성/만료/자동종료, active-user 캐시 동기화, 재시작 시 세션 리셋, 멱등 처리.
- 구현/기술: `rest_api.py`, `robot_manager.py`, `db.py`. 만료 세션 자동 비활성화, 로봇 가용성 검사.
- 선택 이유: 세션 꼬임(유령 세션·중복 로그인) 방지 → DB 부분 유니크 인덱스 + 캐시 동기화로 "1유저=1세션" 보장.
- 관련 파일: `control_service/rest_api.py`, `.../robot_manager.py`, `scripts/db/schema.sql`
- 커밋: "harden REST session lifecycle", "session reset on startup", "session idempotency tests", "automatic deactivation of expired sessions".

**기능 4 — 관제 위치보정 (sim/실물 통합)**
- 무엇: admin_ui에서 로봇 위치를 보정 — 시뮬은 Gazebo teleport, 실물은 AMCL initialpose fallback으로 **단일 명령 통합**.
- 구현/기술: `robot_manager.py` teleport→position_adjustment 리네이밍(호환 유지), teleport bridge 노드, `ros_node.py` 스레딩+타임아웃 서비스 콜.
- 선택 이유: 시뮬/실물이 다른 방식이라 UI 분기 필요 → **하나의 admin 명령으로 추상화**해 운영 단순화.
- 관련 파일: `control_service/robot_manager.py`, `.../ros_node.py`, `admin_ui/robot_card.py`
- 커밋: "unify admin position adjustment for simulation and real robot (AMCL fallback)", "teleport bridge node".

**기능 5 — BoundaryMonitor (결제구역 진입 감지)**
- 무엇: `amcl_pose` 구독해 결제구역 사각 경계 진입 시 콜백 발화(결제 팝업 트리거).

**기능 6 — DB 스키마 + pgvector 상품 임베딩 시딩**

- 구현/기술: `schema.sql`, `seed_data.sql`, `llm_data.sql`, 임베딩 시딩 스크립트, docker-compose DB.
- 선택 이유: 키워드 매칭만으론 자연어 검색 한계 → pgvector 코사인 검색으로 의미 기반 상품 매칭.
- 커밋: "add PRODUCT_TEXT_EMBEDDING table", "add script to sentence embeddings", "restore full PostgreSQL schema and seeds".

- 무엇: customer_web 세션/로그인 관련 파일 스테이징 시 자동 테스트, 실패하면 커밋 차단.

## 2-3. 기술적 어필 포인트

- **난이도 높았던 구현** — 상태머신 전환 무결성: `is_locked_return`/`previous_tracking_state`로 "미결제 잠금 후 복귀해도 세션 유지, 스태프 해제 전엔 종료 불가" 같은 조건부 전환을 안전하게 설계. 임의 진입을 막기 위해 전환 소스를 명시적으로 제한.
