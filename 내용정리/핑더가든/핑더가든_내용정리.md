# PART 1. 프로젝트 전체 설명

## 1-1. 프로젝트 개요

| 항목 | 내용 |
| --- | --- |
| 프로젝트명 | **pingdergarten (핑더가든)** |
| 한 줄 요약(태그라인) | 유치원에서 아이와 놀아주고 교사를 보조하는 **교육보조로봇 서비스** — 3종 로봇이 등원부터 하원까지 하루 일과를 함께한다 |
| 기간 | 2026.04.23 – 2026.06.04 · 약 7주 (실제 커밋 활동 04-28 ~ 06-04) |
| 팀 구성 | **6명** — 로봇별 소프트웨어 분담. 본인은 **GogoPing(Vic Pinky) 전담 리드** + 관제 Admin 앱 (아래 1-2/PART 2 참조) |
| 주요 사용자 / 타깃 | 유치원 **아이**(놀이·상호작용), **교사**(운영·관제·보조), **학부모**(등하원·보고서 조회) |
| 프로젝트 규모 | 로봇 3종 · 사용자 요구사항(UR) 20개 · 시스템 요구사항(SR) **121개**(구현 완료 38 + 계획 83) · 전체 커밋 527 |
| 진행 방식 | 시뮬레이션(Gazebo/MuJoCo) 검증 → 실물 로봇 이전 |
| 이 프로젝트가 증명하는 나의 키워드 | **ROS2 자율주행(Nav2)**, **Behavior Tree/FSM 로봇 행동 설계**, **비전 기반 사람 추종**, **관제 데스크톱 앱**, **테스트 주도 로봇 SW**, **실물 하드웨어 브링업** |
| 관련 링크 | GitHub: `github.com/addinedu-physicalai-1st/physical-ai-repo-2` (조직 비공개) · 배포 URL: 없음(로컬 시연) · 시연 영상: (본인 확인 필요) · 발표자료: `presentation/final-v3/` |

- **프로젝트 주제 / 개요**: 유치원 일과(등원 → 놀이·레크리에이션 → 낮잠 → 하원)를 3종 교육보조로봇이 보조하는 서비스.
  - **EduPing** (= OpenArm 듀얼암): 등하원 응대(얼굴인식 출석·환영/작별), 율동, 가게놀이, 정리정돈, "무궁화꽃이 피었습니다".
  - **GogoPing** (= Vic Pinky 모바일 베이스): 교사 추종·운반 보조, 숨바꼭질, 자장가, 자율 순찰.
  - **NoriArm** (= OMX 매니퓰레이터, 교실 비치): 블럭쌓기, OX 퀴즈, 가게놀이, 정리 — 모방학습(ACT) 기반.
- **기획 배경 / 해결하려는 문제**: 유치원 교사 1인당 돌봐야 하는 아이가 많아 개별 상호작용·반복 업무(출석·운반·놀이 진행) 부담이 큼. 로봇이 정형 업무를 보조하고 아이와의 놀이를 담당해 교사 부담을 줄이는 것이 목표.
- **프로젝트 목표**: 등원~하원 전 구간을 커버하는 **다중 로봇 통합 서비스**를 시뮬레이션에서 검증하고 실물로 이전. 자연어 음성 명령·표정·자율주행·모방학습·얼굴인식을 하나의 서비스 흐름으로 통합.
- **핵심 기능 요약**: 얼굴인식 자동 출석, 자연어 음성 명령/모드 전환, 로봇 표정(three.js 셰이더), 교사 추종·운반, 놀이 다수(무궁화꽃·숨바꼭질·블럭쌓기·가게놀이 등), 학부모 포털(출결·메뉴·사진첩·일과 보고서), 교사용 관제 데스크톱 앱, 카메라 실시간 스트리밍.
- **전체 동작 흐름 (end-to-end)**:
  ```
  [아이/교사 음성] → 로봇 UI(브라우저, Web Speech STT) → 호출어 감지 → Control Service(REST/WS)
       → AI Hub(Ollama 의도 분류) → 모드/서브명령 결정 → Control Service → ROS2(rclpy) 브리지
       → 각 로봇 컨트롤러(Nav2 / MoveIt2 / BT·FSM / ACT 정책) → 구동
       → 로봇 상태 1Hz publish → Control Service → Admin 앱(관제) / 로봇 UI(표정·피드백)
  [카메라] Pi/노트북 → UDP MJPEG → Control Service 스트리밍 GW(8100) → WS fan-out → Admin/웹
  [학부모] Portal Web → Control Service REST → PostgreSQL(출결·메뉴·사진·보고서)
  ```

## 2-2. 내가 구현한 기능 (기능별)

**기능 1 — GogoPing 자율주행 & nav graph 라우터**
- 무엇을: 지도 위 정점(vertex) 그래프를 **Dijkstra 최단경로**로 풀어 Nav2(`FollowWaypoints`/`NavigateThroughPoses`)에 waypoint 시퀀스로 넘기는 라우터. lane 그래프 편집·리로드, 이동 중 목적지 재지정(reroute) 지원.
- 구현/기술: ROS2 노드(`graph_router_node`) + action/srv 정의, Nav2, Smac2D 전역 플래너, `graph.py`(그래프 자료구조·Dijkstra).
- **선택 이유**: 유치원 실내는 자유공간 전 영역 주행보다 **정해진 통로(lane) 위 이동**이 안전·예측가능 → 순수 Nav2 goal 대신 그래프 제약 라우팅. 전역 플래너는 시뮬 튜닝과 일치시키려 기본→**Smac2D로 교체**.
- 관련 파일: `controller/gogoping-controller/src/gogoping/gogoping_navigation/{graph_router_node,graph}.py`, `config/{waypoints,lanes}.default.yaml`.
- 데이터 흐름: (목적지 vertex) → Dijkstra 경로 → waypoint 리스트 → Nav2 action → 주행 → 상태 publish.
- 알고리즘/자료구조: **가중 그래프 + Dijkstra**, ROS action(비동기 goal).
- 구체 근거: `e381516`(그래프 라우터+Dijkstra+action/srv, +572, 테스트 140줄 포함), `bfa4bab`(+269), `7642a1d`/`913c3cc`(Smac2D 전환), `edf8ed2`(주행 중 목적지 리다이렉트).

**기능 2 — Behavior Tree / FSM 모드·행동 시스템**
- 무엇을: py_trees BT + pytransitions FSM(**6 상태·11 트리거**)로 GogoPing의 모든 자율 행동을 구성. 모드별 main tree + 공유 shell, 리프 behavior 다수.
- 구현/기술: `gogoping_modes`(BT trees/behaviors/interfaces), FSM(`robot_fsm.py`).
- **선택 이유**: 숨바꼭질·도킹·순찰처럼 **다단계 조건부 행동**을 순수 코드 if/else로 짜면 상태 폭발 → 선언적 BT로 재사용·시각화·테스트 가능하게. ASSIST/PLAY 통합 트리를 **per-mode 트리로 평탄화**해 복잡도 절감.
- 관련 파일: `gogoping_modes/bt/{trees/main_trees, behaviors/common}`, `gogoping_modes/interfaces`.
- 대표 behavior: 도킹(`AlignToDock`/`ReverseIntoDock`/`VerifyDockingContact`), 숨바꼭질 6-step(카운트다운·caught-monitor·recruit), 순찰(`SelectVertex`/`PanCameraSweep`), 자장가, 수동 토크 홀드, e-stop(`StopAllMotors`).

> 아래 대표 사례 1건은 STAR로, 나머지는 **커밋 근거 트러블슈팅 로그**로 정리. 커밋 본문이 비어 있어 근거는 **커밋 제목**이며, "어떻게 디버깅했는지"의 상세는 본인 기억으로 보완(=`(본인 보완)`).

### 대표 STAR — 추종/perception을 "테스트 가능·안전"하게 재구조화
- **문제 정의 (S·T)**: 추종(FOLLOW)·근접 로직이 `perception_node` 한 곳에 뭉쳐 있어 (1) 하드웨어 없이는 검증 불가, (2) 안전정지 조건 변경이 위험.
- **접근 전략 (A)**: YOLO 추론·proximity·follow 파이프라인·debug overlay를 **순수 모듈로 추출**(`ca2bb3f`, `6c79833`) → 유닛테스트 확보 → 근접/장애물 상태를 ROS→브리지→UI까지 **end-to-end로 노출**(`63ac3af`).
- **의사결정 기준**: 안전 최우선(사람 근접 시 무조건 감속/정지), 실물 없이 회귀 검증 가능.
- **트레이드오프**:
  | 구분 | 대안 A: 단일 perception_node 유지 | 대안 B(선택): 모듈 분리 + 테스트 |
  | --- | --- | --- |
  | 장점 | 빠른 초기 구현 | 테스트·재사용·안전조건 격리 |
  | 단점 | 검증 불가·수정 위험 | 초기 리팩터 비용 |
  - **얻은 것**: 유닛테스트(`tests/` 리포 최다 touch), 안전정지 신뢰성.
  - **감수한 것**: 리팩터 시간, 모듈 경계 설계 비용.
