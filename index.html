Behavior & Interaction03 / 07
FSM + Behavior Tree
MainTree[state] = Parallel(Main 감시 + Sub 행동)
  • FSM=언제 어떤 상태로, BT=그 상태를 어떻게 수행 — 역할 분리
  • Main(monitor) — 상태 무관 병렬로 배터리·맵경계·HW·근접·명령 감시, 이벤트 시 FSM trigger 발화. 항상 RUNNING
  • Sub(body) — 그 상태일 때만 도는 실제 행동. 복잡한 상태(도킹·숨바꼭질·복귀)는 SubTree로 캡슐화
이렇게 나눈 이유
  • 안전·명령 감시는 어떤 상태에서도 멈추면 안 됨 → Main 계층 분리
  • BT 교체 시 root.stop(INVALID)로 안전 종료(cmd_vel=0·goal cancel)
  • 결과 — leaf 26종 → 5개 패턴으로 정리, 유지보수·테스트 용이
숨바꼭질

수동 모드 · 다중 명령
  • ManualTorqueHold — MANUAL 진입 시 토크 OFF로 사람이 직접 로봇을 밀 수 있음, 다른 상태로 나가면 토크 자동 복원

Rerouting & Error Handling04 / 07
경로 재추종
▶경로 재추종 · 2배속 · 클릭 재생

D435 Depth로 사람 근접을 관측(ProximitySafetyMonitor · /proximity_event)하면 즉시 정지(PAUSE) — 사람이 1초 이상 확실히 비키면 원래 경로로 그대로 재개(RESUME)합니다.

비키지 않고 10초 이상 지속되면 그 지점을 막힘(blocked)으로 표시하고 우회 경로를 재탐색(REROUTE)합니다 — 안 비킬 때만 새 경로를 계산합니다.

각종 에러 처리
즉시 정지

긴급 정지 — e-stop → StopAllMotors(ERROR)로 즉시 정지·토크 release. BT 교체 시 terminate로 cmd_vel=0 · goal cancel 강제.

상태 무관 · 항상 도는 감시
  • HardwareHealth — LIDAR·odom 3초 미수신 시 fault(부팅 15초 grace)
  • MapBoundary — OccupancyGrid 기반 맵 밖/unknown 셀 감지 시 fault
  • CollisionMonitor — nav2 collision_monitor 정지가 5분 지속되면 작업 취소(cancel) 또는 fault
  • BatteryLow / Full — 배터리 임계 시 자동 복귀·충전
  • IdleTimeout — IDLE 진입 후 15분 무명령 시 자동 복귀

아이 안전 설계

Child Safety05 / 07
아이 곁을 달리는 로봇의 물리적 위험

유치원 실내에서 아이와 같은 공간을 주행하는 로봇이 만드는 두 가지 안전 위협을, 주행·인식 설계로 제거했습니다.

①

좁은 통로 · 차량 후미 스윙

▶실물 데모
▶경로 시각화

문제 — 구동 바퀴가 앞쪽, 차체는 뒤로 긴 구조라(footprint 앞 +0.10m·뒤 −0.50m) 제자리 회전 시 뒤꽁무니가 반경 0.57m 호를 휩씁니다. 좁은 통로 반폭(~0.5m)을 넘겨 벽은 물론 옆에 있는 아이와 부딪힐 위험이 있었습니다.

해결 — Nav2 표준 Route Server 대신 그래프 라우팅을 커스텀 Dijkstra planner로 직접 구현했습니다. 진행 방향과 135° 넘게 꺾이는 급선회 차선은 Heading-aware Dijkstra가 후보에서 아예 제외하고, 남은 회전은 벽에서 충분히 떨어진 can_rotate 5지점(30개 vertex 중)에서만 허용했습니다.

②

로봇이 높아 아동 미검출

▶Collision Monitor · 클릭 재생

문제 — 로봇 카메라가 고정(축 없음)이면 키가 작은 유치원생이 가까이 다가와도 시야 각도가 안 맞아 놓칠 수 있었습니다.

해결 — D435 Depth 카메라를 2축(pan/tilt) 모터로 움직여 가까이 있는 저신장 아동도 놓치지 않고 보도록 했습니다. 여기에 더해 nav2 collision_monitor 패키지(LiDAR 전방 0.40m·좌우 0.20m 정지구역)까지 함께 써서 2중으로 더 안전하게 대응합니다.

Planner Selection

sim 정량 검증(rosbag)을 통한 우리 맵 환경에 맞는 planner/controller server 선정

Planner Selection06 / 07
DOE 실험 설계 — 3 planner × 3 controller = 9 cells
DOE 결과 — 9 cells × 5 reps = 45 run 측정
문제 상황

실물 테스트 시간·공간이 부족 — 유치원 실증 전 주행 품질을 정량적으로 검증할 방법이 필요했고, 감(느낌)으로 고른 조합은 나중에 근거를 설명할 수 없다는 것도 문제였습니다.

접근 방법

vertex는 실제 벽에서 충분히 떨어지도록 배치해뒀지만, 커스텀 planner가 지정한 노드(vertex) 사이 간선(lane)을 얼마나 딱 붙어 추종하는지는 planner·controller 조합마다 달라 검증이 필요했습니다.

실물 대신 Gazebo에서 rosbag·log 기반 정량 검증 체계를 만들어 3개 planner(NavFn·Smac2D·ThetaStar) × 3개 controller(RPP·DWB·MPPI) = 9 cells를 5회씩 반복(45 run) 비교했습니다. run마다 odom/cmd_vel Hz·freeze 여부를 가중합한 reliability score로 불안정한 run을 걸러내고, 부드러움은 jerk RMS로 정량화했습니다.
